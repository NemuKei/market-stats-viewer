# Purpose
仕様変更の品質を上げるため、実装前に「壁打ち -> 仕様確定 -> タスク化」の順序を固定する。

# When to use / When NOT to use
- When to use:
  - 仕様追加や仕様修正を検討するとき
  - 要件が曖昧で、論点整理や代替案比較が必要なとき
  - すぐ実装する前に、判断漏れを潰したいとき
- When NOT to use:
  - 軽微な文言修正のみで仕様判断を伴わないとき
  - すでに仕様とタスクが確定済みで、実装のみ行うとき

# Inputs
- ユーザーの要望（現状の課題、実現したいこと）
- 既存仕様・制約（必要最小限の関連ドキュメント）

# Outputs
- 壁打ち結果（論点、見落とし補完、上位提案、推奨方針）
- 仕様確定メモ（確定事項・非採用事項）
- `tasks_backlog.md` への実行可能タスク

# Procedure
1. まず壁打ちを行い、目的・制約・成功条件を短く再定義する。
2. 見落としになりやすい観点を補完する（UX、運用、データ整合、例外、回帰影響、検証方法）。
3. 少なくとも1つは上位提案を出し、現行案とのトレードオフを明示する。
4. 未確定論点を列挙し、ユーザー回答で仕様を確定させる。
5. 仕様が確定するまでは実装しない。
6. 仕様確定後、最初に `tasks_backlog.md` へタスク化する。
7. タスクは実装単位で分割し、各タスクに目的・スコープ・受け入れ条件を含める。
8. タスク化完了をユーザーに共有してから実装に進む。

# Validation
- 実装前に壁打ち結果が提示されている。
- 見落とし補完と上位提案の両方が含まれている。
- 仕様確定前にコード変更していない。
- 仕様確定後に `tasks_backlog.md` が更新されている。

# Examples
- 「この仕様追加、先に壁打ちしてからタスク化して」
- 「実装前に抜け漏れチェックと上位案を出して、確定後にタスクへ落として」
