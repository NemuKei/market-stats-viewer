# Purpose
タスク追加のたびにバックログを再整理し、重複や順序崩れを防ぐ。
実装時に迷わない粒度（1タスク=1実装成果が検証可能）へ正規化する。
root `AGENTS.md` の repo-wide ルールを前提にし、この Skill は task add and triage に必要な task-specific procedure だけを追加する。

# When to use / When NOT to use
- When to use:
  - `tasks_backlog.md` に新規タスクを追加した直後
  - 追加タスクが既存タスクと重複・依存しそうなとき
  - 実装順序を `Now/Next` に反映したいとき
- When NOT to use:
  - 単純な typo 修正のみでタスク構造が変わらないとき
  - 完了チェックの更新だけで、優先順位見直しが不要なとき

# Inputs
- 追加したタスク（ID, Objective, Scope, Acceptance）
- 未実装タスク一覧（`- [ ]`）
- 粒度判定に必要な変更面（対象ファイル/層/検証観点）

# Outputs
- 棚卸し結果（重複・統合候補・依存関係）
- 実装粒度へ分割済みタスク（必要時は `P*-NN-01` 形式の子タスク）
- 更新済み `Remaining Task Triage (ASCII)` セクション
- 最適化後の未実装タスク実行順（Now/Next/After Next/Later）
- 必要時のみ、統合後の実行順ルール

# Backlog Formatting Guard
- タスク本体は必ず該当 Phase の本文ブロック（`### タスク`）へ追加/更新する。
- 補助的な機械可読セクションを更新する場合でも、タスク本体を Phase 本文から分離しない。
- 同一 Task ID の重複記載を禁止し、重複が見つかった場合は1箇所へ統合する。
- `Remaining Task Triage` は本文の実装順と整合を保ち、孤立した末尾追記を作らない。
- 実装粒度は「1タスク=1つの主成果物+1つの主検証」で完了判定できる単位を基本とする。
- 1タスクが複数レイヤ（例: parser + GUI + tests）へまたがる場合は、原則として子タスクへ分割する。
- 受け入れ条件は「確認方法が明示できる文」で記述し、抽象語のみ（改善する/対応する）で終わらせない。

# Procedure
1. 新規タスク追加後、未実装タスクを一覧化する。
2. 追加タスクの粒度を判定する（主成果物・対象層・検証方法が1つに収まるか）。
3. 粒度が粗い場合は子タスクへ分割する（例: `P3-73-01` 形式）。親タスクには目的/完了条件のみ残す。
4. 追加タスクと既存タスクの重複・包含関係を確認する。
5. 統合可能なタスク群をエピック単位で抽出する。
6. 依存関係と効果対コストに基づいて、未実装タスクの順番を最適化する。
7. 最適化結果を `Now/Next/After Next/Later` に反映する。
8. `Remaining Task Triage (ASCII)` に統合結果を反映する。
9. 統合した場合は、元タスクIDの追跡可能性を残す（ID列挙または参照）。
10. 更新後、実装対象がどこまでかをユーザーへ明示する。

# Validation
- 新規追加タスクが `Now/Next` のどこに入るか明記されている。
- 追加タスクが実装粒度として過不足ない（粗い場合は子タスク化済み）。
- 重複・統合候補の判断理由がある。
- 並び順変更の理由（依存 or 効果対コスト）が明記されている。
- `Remaining Task Triage (ASCII)` が古い状態のまま残っていない。
- 実装順序ルールが現在のバックログ構造と整合している。
- タスク本体が Phase 本文外へ分離されていない。
- 同一 Task ID が 1 箇所に正規化されている。

# Examples
- 「タスク追加したので、残タスクを棚卸しして統合整理までやって」
- 「追加後にNow/Nextを更新して、重複をまとめて」
