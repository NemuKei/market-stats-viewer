# Purpose
1スレッドで扱う目的と範囲を明確にし、途中で出た別論点を無秩序に混ぜず、必要な場合だけ次スレッド用 handoff へ整理する。

# When to use / When NOT to use
- When to use:
  - handoff prompt から新しいスレッドを始めるとき
  - 実装、仕様壁打ち、レビュー、調査など、複数段階になりうるスレッドを始めるとき
  - 話題が広がりやすく、どこまでこのスレッドで扱うかを先に固定したいとき
- When NOT to use:
  - 単発の短い質問で、その場で回答して終わるとき
  - 明確な thread contract を置くまでもなく、1往復で閉じられる軽微な依頼のとき

# Inputs
- ユーザーの依頼
- 前スレッドの handoff prompt（ある場合）
- 今回のスレッドで必要な最小限の関連コンテキスト

# Outputs
- スレッド冒頭の thread contract
- 途中で発生した parking 論点の整理
- スレッド終了時の状態整理
- ユーザー確認後の handoff prompt（必要な場合のみ）

# Thread contract
スレッド開始時に、少なくとも次を短く明示する。

- `このスレッドの目的`
- `今回の範囲`
- `今回やらないこと`
- `終了条件`
- `終了時に確認が必要なもの`

補足:

- 前スレッドの handoff prompt で始まった場合は、まずその handoff の範囲を今回の `今回の範囲` に写し替える。
- `今回やらないこと` には、関連話題でもこのスレッドへ混ぜないものを書く。

## Thread contract template

```md
このスレッドの目的:
- ...

今回の範囲:
- ...

今回やらないこと:
- ...

終了条件:
- ...

終了時に確認が必要なもの:
- ...
```

# Thread type
必要に応じて、今回のスレッドを次のいずれかに分類して扱う。

- `task-execution`
- `spec-discussion`
- `review`
- `question-answer`
- `handoff-consumption`

分類は厳密なラベル付け自体が目的ではない。終了条件と parking 判断を安定させるために使う。

# Procedure
1. スレッド冒頭で依頼内容を読み、今回の主目的を1つに定める。
2. `今回の範囲` と `今回やらないこと` を切り分ける。
3. `終了条件` を、作業完了条件と確認条件に分けて定義する。
4. 実行中に新しい論点が出たら、現在の主目的に必要かどうかで次のどちらかに分ける。
   - 必要: このスレッドで扱う
   - 不要: parking として整理する
5. parking 論点は、その場で task 化や実装開始へ進めず、次スレッドで再開できる最小整理だけ残す。
6. スレッド終了時に、現在状態を次のいずれかへ分類する。
   - `close-no-handoff`
   - `close-with-handoff-candidate`
   - `close-awaiting-confirmation`
7. handoff prompt は自動で出さない。終了時に必ずユーザーへ確認してから出す。

# Parking rule
現在の主目的に不要な論点は、少なくとも次を残して parking する。

- `話題名`
- `今このスレッドで扱わない理由`
- `決まっていること`
- `未決事項`
- `次スレッドで最初に確認すること`

補足:

- `task-execution` 中に別機能の仕様壁打ちが始まった場合も、原則は parking とする。
- ただし、その壁打ちが現在タスクの完了に直接必要な場合だけ、このスレッドで扱ってよい。

## Parking template

```md
Parking:
- 話題名: ...
- 今このスレッドで扱わない理由: ...
- 決まっていること: ...
- 未決事項: ...
- 次スレッドで最初に確認すること: ...
```

# Completion rule
完了は次の2段階で扱う。

- `作業完了`
  - このスレッドで実施すると決めた作業が終わった状態
- `スレッド完了`
  - `作業完了` に加えて、必要な確認も満たした状態

例:

- テストは通ったが GUI確認が未了
  - `作業完了`
  - `スレッド完了` ではない

# Close states
## `close-no-handoff`
- 今回の範囲が終わり、次スレッドが必須ではない状態

## `close-with-handoff-candidate`
- 今回の範囲は終わったが、次に進めるべき明確なタスクがある状態
- この場合は、次に進める想定タスクを明示したうえで、handoff prompt を作成してよいかをユーザーへ確認する

## `close-awaiting-confirmation`
- GUI確認、ユーザー判断、追加verify などが残っており、スレッド完了に至っていない状態
- この場合は、何の確認待ちかを明示し、同スレッドで続けるか、確認待ち handoff にするかをユーザーへ確認する

## Close summary template

```md
今回の範囲で完了したこと:
- ...

未完了のこと:
- ...

確認済み事項:
- ...

確認待ち事項:
- ...

現在の終了状態:
- close-no-handoff / close-with-handoff-candidate / close-awaiting-confirmation
```

# Handoff confirmation rule
handoff prompt を出す前に、終了時は次のように確認する。

- 次タスクが明確な場合:
  - 「次はこのタスクを進める想定です。次スレッド用 handoff prompt を作成しますか」
- 次タスクが明確でない場合:
  - 「未決事項はこれです。handoff prompt を作るか、このまま閉じるか、追加で整理するかを決めてください」

# Handoff prompt
ユーザーが handoff prompt 作成を希望した場合は、少なくとも次を含める。

- `このスレッドでやったこと`
- `今回の範囲で完了したこと`
- `未完了のこと`
- `次スレッドで最初にやること`
- `今回やらないと決めたこと`
- `確認済み事項`
- `確認待ち事項`
- `参照すべき正本`
- `触ったファイル`
- `再開時の注意点`

## Handoff prompt template

```md
**ハンドオフ**

このスレッドでやったこと
- ...

今回の範囲で完了したこと
- ...

未完了のこと
- ...

次スレッドで最初にやること
- ...

今回やらないと決めたこと
- ...

確認済み事項
- ...

確認待ち事項
- ...

参照すべき正本
- ...

触ったファイル
- ...

再開時の注意点
- ...
```

# Practical guidance
- thread contract は長い計画書にしない。1スレッドで判断と実行を安定させる最小限に留める。
- parking は「あとで考える」の一言で済ませず、次スレッドで再開できる粒度まで整理する。
- 次タスクが明確でも、handoff prompt の自動出力はしない。必ずユーザー確認を挟む。
- GUI確認が未了なら、実装やテストが終わっていても `スレッド完了` とは扱わない。

# Validation
- スレッド冒頭で `目的 / 範囲 / やらないこと / 終了条件` が明示されている
- 途中で広がった論点が、現在スレッドへ含めるものと parking に分かれている
- handoff prompt を自動出力していない
- handoff prompt を出す前にユーザー確認を入れている
- GUI確認やユーザー判断が未了のときに、`スレッド完了` と誤って扱っていない

# Examples
- 「この handoff から再開するときの thread contract を先に決めて」
- 「このスレッドではどこまでやるかを固定して、終わったら handoff 要否を確認して」
- 「途中で別論点が出たら parking して、今スレッドの範囲を守って進めて」
